# LangChain Agent Imports — Interview Q&A Cheat Sheet

## 1. `create_agent`

**Q: What is `create_agent()`?**
**A:** A factory function that creates a complete AI agent capable of reasoning, calling tools, handling memory, and generating responses.

**Q: Why use it?**
**A:** It abstracts the agent loop and orchestration logic, so developers don't need to manually manage tool execution and reasoning.

---

## 2. `ToolStrategy`

**Q: What is `ToolStrategy`?**
**A:** A structured output strategy that forces the model to return data matching a defined schema.

**Q: Why is it useful?**
**A:** It improves reliability and validation compared to asking the model to generate raw JSON.

---

## 3. `tool`

**Q: What does the `@tool` decorator do?**
**A:** It converts a Python function into an agent-callable tool with metadata and argument schemas.

**Q: Why is it needed?**
**A:** Agents can only invoke functions that are registered as tools.

---

## 4. `ToolRuntime`

**Q: What is `ToolRuntime`?**
**A:** An object that provides tools access to runtime context, state, memory, configuration, and stores.

**Q: Why use it?**
**A:** It lets tools access agent information without exposing it as user inputs.

---

## 5. `init_chat_model`

**Q: What is `init_chat_model()`?**
**A:** A provider-agnostic way to initialize chat models.

**Q: What is its benefit?**
**A:** It makes switching between OpenAI, Anthropic, Google, and other providers easier.

---

## 6. `InMemorySaver`

**Q: What is `InMemorySaver`?**
**A:** A checkpoint saver that stores agent state in memory.

**Q: Limitation?**
**A:** Data is lost when the application restarts.

---

## 7. `AIMessage`

**Q: What is `AIMessage`?**
**A:** A standardized object representing a message generated by the AI.

**Q: Why not use plain strings?**
**A:** It supports metadata, tool calls, and structured conversation management.

---

## 8. `wrap_model_call`

**Q: What is `wrap_model_call`?**
**A:** Middleware that intercepts model requests and responses.

**Q: Common use cases?**
**A:** Logging, routing, monitoring, guardrails, and cost tracking.

---

## 9. `dynamic_prompt`

**Q: What is `dynamic_prompt`?**
**A:** Middleware that generates prompts dynamically at runtime.

**Q: Why use it?**
**A:** To customize prompts based on context, memory, or user information.

---

## 10. `wrap_tool_call`

**Q: What is `wrap_tool_call`?**
**A:** Middleware that intercepts tool execution.

**Q: Common use cases?**
**A:** Logging, validation, retries, and access control.

---

## 11. `ModelRequest`

**Q: What is `ModelRequest`?**
**A:** An object containing all information required before sending a request to an LLM.

**Q: What does it contain?**
**A:** Messages, model configuration, state, and metadata.

---

## 12. `ModelResponse`

**Q: What is `ModelResponse`?**
**A:** An object representing the model's output.

**Q: Why use it?**
**A:** It standardizes responses across different LLM providers.

---

## 13. `dataclass`

**Q: What is `@dataclass`?**
**A:** A Python decorator that automatically generates boilerplate code for data containers.

**Q: Why is it common in LangChain?**
**A:** It's often used to define structured output schemas.

---

## 14. `ToolMessage`

**Q: What is `ToolMessage`?**
**A:** A message object containing the output of a tool execution.

**Q: Why is it important?**
**A:** It allows the model to receive and reason over tool results.

---

## 15. `ToolCallRequest`

**Q: What is `ToolCallRequest`?**
**A:** An object representing a request to execute a tool.

**Q: What does it contain?**
**A:** Tool name, arguments, runtime context, and execution metadata.

---
